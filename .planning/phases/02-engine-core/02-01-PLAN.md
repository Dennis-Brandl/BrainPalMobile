    **Create EngineEventBus** (`src/events/event-bus.ts`):
    - Typed EventEmitter using EngineEventMap from types/events.ts
    - `on<K>(event, handler)` returns unsubscribe function
    - `emit<K>(event, data)` calls all registered handlers
    - `removeAllListeners()` clears all handlers
    - Use Map<string, Set<Function>> internally (zero external dependencies)
    - Follow exact pattern from research Pattern 4

    **Create EngineEventQueue** (`src/events/event-queue.ts`):
    - Serial async FIFO queue -- CRITICAL for preventing race conditions in parallel branches
    - Constructor takes `handler: (event: EngineEvent) => Promise<void>`
    - `enqueue(event)` returns Promise<void> that resolves when handler completes for that event
    - `processNext()` processes events one at a time; if already processing, new events wait in queue
    - Error in handler rejects the enqueue promise but does not stop queue processing
    - Follow exact pattern from research Pattern 1
    - NOTE: The EngineEvent type used here should be the discriminated union from events.ts (or a generic type param if cleaner)

    **Create event barrel** (`src/events/index.ts`): Re-export bus and queue.

    **Create mock repositories** (`__tests__/helpers/mock-repositories.ts`):
    - InMemoryWorkflowRepository implementing IWorkflowRepository (uses Map<string, RuntimeWorkflow>)
    - InMemoryStepRepository implementing IStepRepository (uses Map<string, RuntimeWorkflowStep>)
    - InMemoryConnectionRepository implementing IConnectionRepository
    - InMemoryValuePropertyRepository implementing IValuePropertyRepository
    - InMemoryMasterWorkflowRepository implementing IMasterWorkflowRepository
    - InMemoryExecutionLogger implementing IExecutionLogger (stores entries in array)
    - TestIdGenerator implementing IIdGenerator (returns sequential UUIDs like "test-id-1", "test-id-2")
    - Each mock uses Map/Array internally, implementing all interface methods
    - Add a `clear()` method to each mock for test cleanup

    **Create test fixtures** (`__tests__/helpers/fixtures.ts`):
    - `makeLinearWorkflow(stepCount)` -- creates MasterWorkflowSpecification with START -> N user_interaction steps -> END, properly connected
    - `makeParallelWorkflow()` -- creates START -> PARALLEL -> [branch A steps, branch B steps] -> WAIT_ALL -> END
    - `makeSelect1Workflow(branchCount)` -- creates START -> USER_INTERACTION -> SELECT_1 -> [N branches] -> END
    - Each fixture creates well-formed specs with oids, local_ids, connections, and minimal parameters
    - These are used by tests in plans 02-02 through 02-05

    **Create test utils** (`__tests__/helpers/test-utils.ts`):
