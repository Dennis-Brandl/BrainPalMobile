    **2. Create notification service (mobile)** (`apps/mobile/src/services/notification-service.ts`):
    - IMPORTANT: This file handles MOBILE notifications only. Use platform-specific files to avoid crashing web.
    - Create `notification-service.ts` (mobile) and `notification-service.web.ts` (web). Metro/bundler resolves `.web.ts` for web platform and `.ts` for native.
    - Mobile service (`notification-service.ts`):
      ```
      import * as Notifications from 'expo-notifications';
      import * as Device from 'expo-device';
      ```
      - `NotificationService` class with:
        - `constructor(private db: SQLiteDatabase)` -- needs DB for preference lookups
        - `async initialize()`:
          a. If not physical device (`!Device.isDevice`), skip notification setup (emulator may not support)
          b. Request permissions: `await Notifications.requestPermissionsAsync()`
          c. Set up Android notification channels:
            - 'step-attention': name "Step Needs Attention", importance HIGH, vibration, sound
            - 'errors': name "Errors", importance MAX, vibration, sound
          d. Set notification handler for foreground:
            ```
            Notifications.setNotificationHandler({
              handleNotification: async () => ({
                shouldShowAlert: true,
                shouldPlaySound: true,
                shouldSetBadge: false,
                shouldShowBanner: true,
                shouldShowList: true,
              }),
            });
            ```
        - `async sendStepAttention(workflowInstanceId: string, stepInstanceId: string, stepName: string)`:
          a. Check preference: query `SELECT enabled FROM notification_preferences WHERE notification_type = 'STEP_ATTENTION'`
          b. If not enabled, return
          c. Schedule notification:
            ```
            await Notifications.scheduleNotificationAsync({
              content: {
                title: 'Step Needs Attention',
                body: stepName || 'A step requires your input',
                data: { workflowInstanceId, stepInstanceId },
                channelId: 'step-attention',
              },
              trigger: null, // immediate
            });
            ```
        - `async sendError(source: string, message: string)`:
          a. Check preference for 'ERROR' type
          b. Schedule notification with channelId 'errors'
        - `setupNotificationTapHandler(router: any)`:
          a. `Notifications.addNotificationResponseReceivedListener((response) => { ... })`
          b. Extract `workflowInstanceId` from `response.notification.request.content.data`
          c. Navigate: `router.push(\`/execution/${data.workflowInstanceId}\`)`
          d. Return the subscription for cleanup
      - Export: `createNotificationService(db: SQLiteDatabase): NotificationService`

    - **3. Web service (`notification-service.web.ts`)** -- MUST implement tap-to-navigate:
      - Same interface as mobile service (NotificationService class with same method signatures)
      - `async initialize()`:
        a. Guard: `if (!('Notification' in window)) return`
        b. Call `Notification.requestPermission()` if available
      - `async sendStepAttention(workflowInstanceId: string, stepInstanceId: string, stepName: string)`:
        a. Check preference from DB (same query as mobile)
        b. If not enabled, return
        c. If `document.hidden` (tab backgrounded): create browser notification:
          ```
          const notification = new Notification('Step Needs Attention', {
            body: stepName || 'A step requires your input',
          });
          notification.onclick = () => {
            window.focus();
            window.location.href = `/execution/${workflowInstanceId}`;
          };
          ```
        d. If tab active: no-op for now (toast can be added later -- out of scope for v1)
      - `async sendError(source: string, message: string)`:
        a. Check preference from DB
        b. If not enabled, return
        c. If `document.hidden`: create browser notification with onclick that focuses window
      - `setupNotificationTapHandler(router: any)`:
        a. **MUST implement onclick navigation**, not a no-op
        b. For web, the onclick handler is set on each individual Notification instance in `sendStepAttention`/`sendError` above, so this method just returns a no-op cleanup function (the onclick is already wired per-notification)
        c. Return `{ remove: () => {} }` for consistent cleanup interface
      - Guard all methods with `if (!('Notification' in window)) return`
      - Export: `createNotificationService(db: SQLiteDatabase): NotificationService`

    **4. Create useNotificationPrefs hook** (`apps/mobile/src/hooks/useNotificationPrefs.ts`):
    - `useNotificationPrefs()` returns `{ prefs: NotificationPref[], loading: boolean, toggle: (type: string, enabled: boolean) => Promise<void> }`
    - Type `NotificationPref` = `{ type: string, label: string, enabled: boolean }`
    - On mount: `SELECT * FROM notification_preferences` and map to prefs array
    - Map notification_type to human-readable labels. Show only user-facing types:
      - 'STEP_ATTENTION' -> 'Step needs attention'
      - 'ERROR' -> 'Errors and failures'
    - **IMPORTANT regarding TIMEOUT type:** The `notification_preferences` table seeds a 'TIMEOUT' row, but the engine does NOT currently emit any TIMEOUT event (there is no 'TIMEOUT' event in `EngineEventMap` in `packages/engine/src/types/events.ts`). Do NOT display the TIMEOUT preference toggle in the Settings UI -- it would be misleading since no timeout notifications can actually fire. Filter it out from the displayed preferences. Leave the seeded row in the database (it can be enabled in a future version when timeout events are implemented).
    - Skip non-user-facing types: ACTION_COMPLETED, STATE_TRANSITION, RESOURCE_ACQUIRED, TIMEOUT
    - `toggle(type, enabled)`: Use upsert pattern (NOT INSERT OR REPLACE):
      ```sql
      INSERT INTO notification_preferences (notification_type, enabled)
      VALUES (?, ?)
      ON CONFLICT(notification_type) DO UPDATE SET enabled = excluded.enabled
      ```
    - After toggle, refresh prefs state

    **5. Create useStorageCounts hook** (`apps/mobile/src/hooks/useStorageCounts.ts`):
    - `useStorageCounts()` returns `{ counts: StorageCounts | null, loading: boolean, refresh: () => void }`
    - Type `StorageCounts` = `{ downloaded: number, active: number, completed: number }`
    - Three parallel queries via Promise.all:
      a. Downloaded: `SELECT COUNT(*) as count FROM master_workflows`
      b. Active: `SELECT COUNT(*) as count FROM runtime_workflows WHERE workflow_state IN ('IDLE', 'RUNNING', 'PAUSED') AND parent_workflow_instance_id IS NULL`
      c. Completed: `SELECT COUNT(*) as count FROM runtime_workflows WHERE workflow_state IN ('COMPLETED', 'ABORTED', 'STOPPED') AND parent_workflow_instance_id IS NULL`
    - Include `refresh()` callback for re-querying after clear operations
