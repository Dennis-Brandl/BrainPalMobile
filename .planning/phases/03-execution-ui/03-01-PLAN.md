**3. Create SQLite repository implementations:**

Each repository class takes `SQLiteDatabase` in the constructor and implements the corresponding engine interface. All repos go in `apps/mobile/src/repositories/`.

Use `db.runAsync()` for INSERT/UPDATE/DELETE and `db.getFirstAsync<RowType>()` / `db.getAllAsync<RowType>()` for SELECT queries. Follow the exact expo-sqlite v16 API patterns already used in the existing stores.

Each repository converts between SQLite row types and engine runtime types. Key conversion patterns:
- `entries_json` (string in SQLite) <-> `entries` (PropertyEntry[] in engine) via JSON.parse/stringify
- `named_instances_json` <-> `named_instances` via JSON.parse/stringify
- NULL columns map to `null` in TypeScript
- Uint8Array for image data stored as BLOB in SQLite

Repository implementations (each in its own file):

- **SqliteMasterWorkflowRepository** implements IMasterWorkflowRepository: getAll reads master_workflows table, getByOid by oid, save inserts with specification_json, deleteByOid deletes by oid, replaceByOid does DELETE + INSERT.

- **SqliteMasterEnvironmentRepository** implements IMasterEnvironmentRepository: getByWorkflowOid, save, deleteByWorkflowOid -- uses master_environments table.

- **SqliteMasterActionRepository** implements IMasterActionRepository: getByEnvironmentOid, save, deleteByEnvironmentOid -- uses master_actions table.

- **SqliteImageRepository** implements IImageRepository: getByWorkflowOid, save (INSERT image data as BLOB), deleteByWorkflowOid -- uses package_images table.

- **SqliteWorkflowRepository** implements IWorkflowRepository: getById, save (INSERT OR REPLACE), updateState, getActive (WHERE workflow_state IN ('IDLE','RUNNING','PAUSED')), delete, updateLastActivity.

- **SqliteStepRepository** implements IStepRepository: getByWorkflow, getById, updateState, save (INSERT OR REPLACE), saveMany (loop save), updateUserInputs (UPDATE user_inputs_json), updateResolvedOutputs.

- **SqliteConnectionRepository** implements IConnectionRepository: getByWorkflow, saveMany (INSERT each connection).

- **SqliteValuePropertyRepository** implements IValuePropertyRepository:
  - getWorkflowProperty: SELECT from workflow_value_properties WHERE workflow_instance_id AND property_name, parse entries_json to PropertyEntry[].
  - getEnvironmentProperty: SELECT from environment_value_properties WHERE environment_oid AND property_name.
  - upsertEntry: Read existing property, merge entry (update or append), write back entries_json. Use INSERT OR REPLACE. Handle both workflow and environment scope tables.
  - deleteByWorkflow: DELETE FROM workflow_value_properties WHERE workflow_instance_id.
  - initializeFromSpec: For each PropertySpecification, INSERT entries_json from spec entries.
  - getAllByWorkflow: SELECT all from workflow_value_properties for the instance.

- **SqliteResourcePoolRepository** implements IResourcePoolRepository: create (INSERT), getByScope, updateUsage, delete.

- **SqliteResourceQueueRepository** implements IResourceQueueRepository: enqueue (INSERT with resource_pool_id lookup), dequeue (SELECT + DELETE with ORDER BY requested_at ASC LIMIT 1), getByPool.

- **SqliteSyncBarrierRepository** implements ISyncBarrierRepository: register (INSERT), match (UPDATE matched_with_step_id and matched_at), getUnmatched (SELECT WHERE matched_with_step_id IS NULL AND resource_name AND command_type). Note: the `id` field in SyncBarrierEntry is a string from the engine's IIdGenerator, but the SQLite table uses INTEGER AUTOINCREMENT. Store the engine's `id` as a separate field or use a mapping. Simplest: add the engine's barrier entry ID as part of the query pattern. Since the schema uses auto-increment, map `register` to INSERT with the engine's string id stored alongside, and `match` uses the internal auto-increment id. Actually, re-examine the interface: `match(entryId: string, matchedWithStepInstanceId: string)` -- the `entryId` is the engine-generated string ID. But the schema doesn't have a TEXT id column for this. **Resolution**: The schema's sync_barriers table needs to work with the engine's string IDs. Store the engine `id` in a column (the schema doesn't have one). Rather than modifying the schema, use the `step_instance_id` + `resource_name` + `command_type` as the lookup key for `match`. The `entryId` from the engine can be the auto-increment `id` cast to string, OR we can use the step_instance_id as the match key. **Best approach**: Use the auto-increment `id` from SQLite as the `SyncBarrierEntry.id` field (cast to string on read, parse to int on write for match). This avoids schema changes and works because the engine just needs a unique identifier.

- **SqliteExecutionLoggerRepository** implements IExecutionLogger: log(entry) does INSERT INTO execution_log_entries.

- **IdGenerator**: Simple class implementing IIdGenerator with `generate(): string` that returns a UUID. Use `crypto.randomUUID()` (available in React Native Hermes engine and web).

Create `apps/mobile/src/repositories/index.ts` barrel export of all repository classes.

**1. Create execution store** (`apps/mobile/src/stores/execution-store.ts`):

Zustand store that maintains the active execution's reactive state for UI rendering. This is the bridge between the engine's event bus and React components.

State shape:
```typescript
interface ExecutionState {
  // Active workflow tracking
  activeWorkflows: Map<string, ActiveWorkflowState>;  // instanceId -> state
  currentWorkflowId: string | null;  // currently viewed workflow

  // Actions
  setCurrentWorkflow: (id: string | null) => void;
  addActiveWorkflow: (instanceId: string, masterOid: string, name: string) => void;
  removeActiveWorkflow: (instanceId: string) => void;
  updateWorkflowState: (instanceId: string, state: WorkflowState) => void;
  updateStepState: (instanceId: string, stepOid: string, state: StepState) => void;
  setActiveSteps: (instanceId: string, stepInstanceIds: string[]) => void;
  loadFromDb: (db: SQLiteDatabase) => Promise<void>;
}

interface ActiveWorkflowState {
  instanceId: string;
  masterOid: string;
  name: string;
  workflowState: WorkflowState;
  stepStates: Map<string, StepState>;  // stepOid -> state
  activeStepInstanceIds: string[];  // for carousel
  currentStepIndex: number;
  totalSteps: number;
  startedAt: string | null;
  lastActivityAt: string | null;
}
```

Use `create` from zustand. Use Maps stored as plain objects (Zustand doesn't deep-compare Maps, so use `new Map(Object.entries(...))` pattern or store as Record<string, T>). Actually, for Zustand compatibility, store `activeWorkflows` as `Record<string, ActiveWorkflowState>` not Map, and `stepStates` as `Record<string, StepState>`.

The `loadFromDb` method queries `runtime_workflows WHERE workflow_state IN ('IDLE', 'RUNNING', 'PAUSED')` and populates activeWorkflows. For each active workflow, also query runtime_steps to populate stepStates and identify active steps (steps in EXECUTING state).

**2. Create EngineProvider** (`apps/mobile/src/providers/EngineProvider.tsx`):

React context that provides the WorkflowRunner to the component tree.

```typescript
interface EngineContextValue {
  runner: WorkflowRunner;
  isReady: boolean;
}
```

The provider:
1. Uses `useSQLiteContext()` to get the database handle
2. Creates all SQLite repository instances (from Task 1)
3. Creates an `EngineEventBus` instance
4. Creates an `ExecutionLogService` with the SqliteExecutionLoggerRepository
5. Creates an `IdGenerator` instance
6. Constructs `RunnerConfig` with all repositories + eventBus + idGenerator + executionLogger
7. Creates `WorkflowRunner` with the config
8. Subscribes the execution store to EngineEventBus events on mount:
   - `WORKFLOW_STARTED` -> `updateWorkflowState(id, 'RUNNING')`
   - `WORKFLOW_COMPLETED` -> `updateWorkflowState(id, 'COMPLETED')`, then `removeActiveWorkflow(id)` after short delay
   - `WORKFLOW_ABORTED` -> `updateWorkflowState(id, 'ABORTED')`, then `removeActiveWorkflow(id)`
   - `WORKFLOW_STOPPED` -> `updateWorkflowState(id, 'STOPPED')`
   - `STEP_STATE_CHANGED` -> `updateStepState(workflowId, stepOid, toState)`
   - `ACTIVE_STEPS_CHANGED` -> `setActiveSteps(workflowId, activeSteps)`
   - `USER_INPUT_REQUIRED` -> (no-op for now, carousel will react to ACTIVE_STEPS_CHANGED)
9. Runs crash recovery on mount: calls `recoverWorkflows(runner, workflowRepo)` imported from engine. For each recovered workflow, call `addActiveWorkflow` on the execution store.
10. Cleans up event bus subscriptions on unmount
11. Uses `useRef` to create runner ONCE (not on every render). Use `useMemo` or `useRef` + initialization flag.

Export `EngineProvider` component and `useEngine()` hook (useContext wrapper).
