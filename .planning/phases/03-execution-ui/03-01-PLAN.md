**1. Create execution store** (`apps/mobile/src/stores/execution-store.ts`):

Zustand store that maintains the active execution's reactive state for UI rendering. This is the bridge between the engine's event bus and React components.

State shape:
```typescript
interface ExecutionState {
  // Active workflow tracking
  activeWorkflows: Map<string, ActiveWorkflowState>;  // instanceId -> state
  currentWorkflowId: string | null;  // currently viewed workflow

  // Actions
  setCurrentWorkflow: (id: string | null) => void;
  addActiveWorkflow: (instanceId: string, masterOid: string, name: string) => void;
  removeActiveWorkflow: (instanceId: string) => void;
  updateWorkflowState: (instanceId: string, state: WorkflowState) => void;
  updateStepState: (instanceId: string, stepOid: string, state: StepState) => void;
  setActiveSteps: (instanceId: string, stepInstanceIds: string[]) => void;
  loadFromDb: (db: SQLiteDatabase) => Promise<void>;
}

interface ActiveWorkflowState {
  instanceId: string;
  masterOid: string;
  name: string;
  workflowState: WorkflowState;
  stepStates: Map<string, StepState>;  // stepOid -> state
  activeStepInstanceIds: string[];  // for carousel
  currentStepIndex: number;
  totalSteps: number;
  startedAt: string | null;
  lastActivityAt: string | null;
}
```

Use `create` from zustand. Use Maps stored as plain objects (Zustand doesn't deep-compare Maps, so use `new Map(Object.entries(...))` pattern or store as Record<string, T>). Actually, for Zustand compatibility, store `activeWorkflows` as `Record<string, ActiveWorkflowState>` not Map, and `stepStates` as `Record<string, StepState>`.

The `loadFromDb` method queries `runtime_workflows WHERE workflow_state IN ('IDLE', 'RUNNING', 'PAUSED')` and populates activeWorkflows. For each active workflow, also query runtime_steps to populate stepStates and identify active steps (steps in EXECUTING state).

**2. Create EngineProvider** (`apps/mobile/src/providers/EngineProvider.tsx`):

React context that provides the WorkflowRunner to the component tree.

```typescript
interface EngineContextValue {
  runner: WorkflowRunner;
  isReady: boolean;
}
```

The provider:
1. Uses `useSQLiteContext()` to get the database handle
2. Creates all SQLite repository instances (from Tasks 1a and 1b)
3. Creates an `EngineEventBus` instance
4. Creates an `ExecutionLogService` with the SqliteExecutionLoggerRepository
5. Creates an `IdGenerator` instance
6. Constructs `RunnerConfig` with all repositories + eventBus + idGenerator + executionLogger
7. Creates `WorkflowRunner` with the config
8. Subscribes the execution store to EngineEventBus events on mount:
   - `WORKFLOW_STARTED` -> `updateWorkflowState(id, 'RUNNING')`
   - `WORKFLOW_COMPLETED` -> `updateWorkflowState(id, 'COMPLETED')`, then `removeActiveWorkflow(id)` after short delay
   - `WORKFLOW_ABORTED` -> `updateWorkflowState(id, 'ABORTED')`, then `removeActiveWorkflow(id)`
   - `WORKFLOW_STOPPED` -> `updateWorkflowState(id, 'STOPPED')`, then `removeActiveWorkflow(id)` (runner.stop() deletes from activeWorkflows, so the UI should mirror this — stopped workflows are terminal in v1)
   - `STEP_STATE_CHANGED` -> `updateStepState(workflowId, stepOid, toState)`. **Also derive active steps from STEP_STATE_CHANGED events** (see below).
   - `USER_INPUT_REQUIRED` -> (no-op for now, active steps are derived from STEP_STATE_CHANGED)

   **ACTIVE_STEPS_CHANGED derivation (important):**
   The `ACTIVE_STEPS_CHANGED` event is defined in the engine's EngineEventMap but is never emitted by WorkflowRunner or StepExecutor. Rather than modifying the engine (Phase 2 code), derive active steps reactively from `STEP_STATE_CHANGED` events:
   - On each `STEP_STATE_CHANGED` event, check `toState`:
     - If `toState` is `'EXECUTING'` and the step type is `USER_INTERACTION` or `YES_NO`: add the stepInstanceId to `activeStepInstanceIds`
     - If `toState` is `'COMPLETING'`, `'COMPLETED'`, `'ABORTED'`, `'PAUSED'`, or any non-EXECUTING state: remove the stepInstanceId from `activeStepInstanceIds`
   - To determine step type from a STEP_STATE_CHANGED event, query the step from SQLite (`stepRepo.getById(stepInstanceId)`) and check `step_type`. Cache the step type in the execution store to avoid repeated queries.
   - This approach is equivalent to subscribing to ACTIVE_STEPS_CHANGED and works with the existing engine emission patterns.
9. Runs crash recovery on mount:
   - Build the `RunnerConfig` object (same config used to construct the runner)
   - Call `recoverWorkflows(config)` imported from `@brainpal/engine` (signature: `recoverWorkflows(config: RunnerConfig): Promise<RecoveryResult>`)
   - **Critical**: `recoverWorkflows` builds `WorkflowRunnerState` internally but does NOT restore it into the runner's `activeWorkflows` map. After the call returns, iterate `result.recovered` and for each recovered workflow ID:
     1. Load the workflow from SQLite: `config.workflowRepo.getById(id)`
     2. Load steps: `config.stepRepo.getByWorkflow(id)`
     3. Load connections: `config.connectionRepo.getByWorkflow(id)`
     4. Rebuild `WorkflowRunnerState` (stateMachines, schedulerContext, stepOidToInstanceId, stepInstanceIdToOid) — same rebuild logic as in crash-recovery.ts
     5. Call `runner.restoreWorkflowState(state)` to register the workflow back into the runner's in-memory `activeWorkflows` Map
   - For each recovered workflow, also call `addActiveWorkflow` on the execution store
   - **Alternative (simpler)**: Extract the state-building into a shared helper or modify crash-recovery.ts to accept the runner and call `restoreWorkflowState` internally. Either approach works — the key requirement is that after recovery, `runner.getActiveWorkflows()` contains all recovered workflows.
10. Cleans up event bus subscriptions on unmount
11. Uses `useRef` to create runner ONCE (not on every render). Use `useMemo` or `useRef` + initialization flag.

Export `EngineProvider` component and `useEngine()` hook (useContext wrapper).
