  <action>
    **Create scheduler types** (`src/scheduler/types.ts`):
    - `AdjacencyList`: Map<string, string[]> (step_oid -> array of connected step_oids)
    - `SchedulerContext`: { outgoing: AdjacencyList, incoming: AdjacencyList, steps: Map<string, RuntimeWorkflowStep>, connections: WorkflowConnection[] }

    **Create Scheduler class** (`src/scheduler/scheduler.ts`):

    `buildAdjacencyLists(connections: WorkflowConnection[])`:
    - Build outgoing map: from_step_oid -> [to_step_oid, ...]
    - Build incoming map: to_step_oid -> [from_step_oid, ...]
    - Return both maps. Build once per workflow creation, reuse throughout execution.

    `getNextSteps(completedStepOid: string, context: SchedulerContext)`:
    - Look up outgoing connections from completedStepOid
    - For each target step, evaluate based on target step type:
      - **Normal step** (START, END, USER_INTERACTION, YES_NO, ACTION_PROXY, SCRIPT, WORKFLOW_PROXY, SELECT_1): Activate directly (return in result)
      - **PARALLEL**: Activate the PARALLEL step itself. When the PARALLEL step completes (handled separately), activate ALL its outgoing connections simultaneously
      - **WAIT_ALL**: Check ALL incoming steps. Only activate if EVERY incoming step is in COMPLETED state. If not all complete, return empty for this step (it waits).
      - **WAIT_ANY**: Check incoming steps. Activate if ANY ONE incoming step is in COMPLETED state. On first trigger, activate. On subsequent triggers, ignore (step already activated).
    - Return array of step_oids to activate

    `getParallelBranchSteps(parallelStepOid: string, context: SchedulerContext)`:
    - Return ALL outgoing connections from the PARALLEL step -- these are the branch start steps
    - Used when PARALLEL step reaches COMPLETED to activate all branches

    Important edge cases:
    - SELECT_1 step: only one outgoing connection is followed (based on condition evaluation result -- the runner passes the matched connection, not the scheduler). The scheduler just follows the single provided connection.
    - END step: no outgoing connections. Return empty array. Workflow completion is handled by the runner.
    - Orphan detection: if a step has no outgoing connections and is not END type, log a warning (but don't crash).

    **Create barrel export** (`src/scheduler/index.ts`).

    **Create scheduler tests** (`__tests__/scheduler/scheduler.test.ts`):
    Using fixtures from `__tests__/helpers/fixtures.ts` and mock repositories:

    - Test `buildAdjacencyLists`: verify outgoing and incoming maps for a linear 3-step workflow
    - Test linear flow: START completes -> activates Step1, Step1 completes -> activates Step2, Step2 completes -> activates END
    - Test PARALLEL fork: PARALLEL step completes -> activates ALL branch start steps (verify 2+ steps returned)
    - Test WAIT_ALL join:
      - Branch A completes but Branch B not yet -> returns empty (no activation)
      - Branch B also completes -> returns WAIT_ALL step (now all incoming are COMPLETED)
    - Test WAIT_ANY join:
      - Branch A completes -> returns WAIT_ANY step immediately
      - Branch B completes later -> returns empty (WAIT_ANY already activated, do not double-activate)
    - Test END step: returns empty array (no next steps)
    - Test no outgoing connections for non-END step: returns empty with warning
