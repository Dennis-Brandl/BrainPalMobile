    **1. Implement WORKFLOW_PROXY case in step-executor.ts** (`packages/engine/src/runner/step-executor.ts`):
    - Replace the `throw new UnsupportedStepTypeError(...)` in the WORKFLOW_PROXY case with:
      a. Guard: assert `ctx.runner` and `ctx.workflowRepo` are defined (throw Error if not -- they must be wired by WorkflowRunner for WORKFLOW_PROXY steps)
      b. Get the parent workflow's full specification_json: `const parentWorkflow = await ctx.workflowRepo.getById(ctx.workflowInstanceId)`, then parse `JSON.parse(parentWorkflow.specification_json)` as `MasterWorkflowSpecification`
      c. **CRITICAL -- Child workflow matching strategy:** The `MasterWorkflowStep` type (which extends ManagedElement) has `oid`, `local_id`, and `description` fields. The `MasterWorkflowSpecification.child_workflows[]` is an array of `MasterWorkflowSpecification` objects (also extending ManagedElement with their own `oid`, `local_id`, `description`). The spec does NOT define an explicit linking field on the step. Use this resolution strategy:
         - If `parentSpec.child_workflows.length === 1`: use that child directly (most common case)
         - If multiple children: try matching `masterStep.description` against child workflow `local_id` (case-insensitive, trimmed). The WORKFLOW_PROXY step's description is typically set to the name of the child workflow it invokes.
         - If still no match: try matching `masterStep.local_id` against child workflow `local_id`
         - If still no match: fall back to positional matching -- find the index of this WORKFLOW_PROXY step among all WORKFLOW_PROXY steps in the parent spec (in step array order) and use the same index into `child_workflows[]`
         - If no child found: throw Error with descriptive message listing available child workflow local_ids
      d. Call `const childInstanceId = await ctx.runner.createChildWorkflow(childSpec, ctx.workflowInstanceId, ctx.step.step_oid)`
      e. Save child reference on parent step: `ctx.step.child_workflow_instance_id = childInstanceId; await ctx.stepRepo.save(ctx.step)`
      f. Start the child: `await ctx.runner.startWorkflow(childInstanceId)`
      g. Return `false` -- parent step stays in EXECUTING while child runs

    **2. Add `createChildWorkflow` method to WorkflowRunner** (`packages/engine/src/runner/workflow-runner.ts`):

      a. Call existing `this.createWorkflow(childSpec)` to get the child instance ID
      b. Fetch the child workflow: `const childWf = await this.config.workflowRepo.getById(childInstanceId)`
      c. Set parent references: `childWf.parent_workflow_instance_id = parentWorkflowInstanceId; childWf.parent_step_oid = parentStepOid`
      d. Save: `await this.config.workflowRepo.save(childWf)`
      e. Return the child instance ID

    **3. Add child completion -> parent resumption logic** (`packages/engine/src/runner/workflow-runner.ts`):
    - In `onStepCompleted`, when an END step completes, BEFORE the existing `completeWorkflow` call:
      - Fetch the workflow: `const workflow = await this.config.workflowRepo.getById(workflowInstanceId)`
      - Check `workflow.parent_workflow_instance_id`
      - If it has a parent (this is a child completing):
        a. **CRITICAL ORDER -- Propagate outputs BEFORE completing child workflow.** `completeWorkflow()` calls `valuePropertyRepo.deleteByWorkflow()` which destroys the child's Value Properties. So output propagation MUST happen first:
           - Read child workflow's output Value Properties via `this.config.valuePropertyRepo.getByScope('workflow', workflowInstanceId)`
           - Store them as JSON in the parent step's `resolved_outputs_json` field
           - The parent step is found via: `const parentRunnerState = this.activeWorkflows.get(workflow.parent_workflow_instance_id)`, then `parentRunnerState.stepOidToInstanceId.get(workflow.parent_step_oid)` to get parent step instance ID, then `this.config.stepRepo.getById(parentStepInstanceId)` to load it
           - Set `parentStep.resolved_outputs_json = JSON.stringify(outputProperties)` and save
        b. Now complete the child workflow (existing `completeWorkflow` call -- this deletes Value Properties, which is fine since we already propagated)
        c. Emit `WORKFLOW_COMPLETED` for the child
        d. Directly call `this.handleUserInputCompletion(workflow.parent_workflow_instance_id, workflow.parent_step_oid, parentStepInstanceId)` for the parent step (EXECUTING -> COMPLETING -> COMPLETED). Use DIRECT call, NOT `eventQueue.enqueue`, because we are already inside the event queue handler. This avoids Pitfall 3 from research.
        e. Remove child from `activeWorkflows`
        f. Return early (do NOT process parent workflow scheduling -- that's handled by the parent step's completion flow)
      - If no parent: proceed with existing logic (completeWorkflow, emit WORKFLOW_COMPLETED, delete from activeWorkflows)

    **4. Add pause/abort/resume/stop propagation** (`packages/engine/src/runner/workflow-runner.ts`):

    - In `resumeWorkflow`: Similarly after resuming PAUSED steps, check WORKFLOW_PROXY steps and recursively call `await this.resumeWorkflow(step.child_workflow_instance_id)`
    - In `abort`: BEFORE aborting parent steps, iterate and find WORKFLOW_PROXY steps with child_workflow_instance_id. For each, recursively call `await this.abort(step.child_workflow_instance_id)` FIRST (so child is cleaned up before parent). Then proceed with existing parent abort logic.
    - In `stop`: Similarly propagate to child before stopping parent steps
