    **Create runner types** (`src/runner/types.ts`):
    - `RunnerConfig`: { repositories (all I* interfaces), eventBus, idGenerator, logger }

    **Create lifecycle functions** (`src/runner/lifecycle.ts`):

    `createRuntimeWorkflow(masterSpec, idGenerator)`:
    - Deep copy master spec via JSON.parse(JSON.stringify(...))
    - Generate workflow instance_id
    - Create RuntimeWorkflow object (state: IDLE, timestamps, null parent fields)
    - Create RuntimeWorkflowStep for each step (state: IDLE, step_json from spec copy)
    - Create WorkflowConnection array from spec connections
    - Return { workflow, steps, connections }
    - Follow exact pattern from research Pattern 5

    `completeWorkflow(workflowInstanceId, repositories)`:
    - Set workflow state to COMPLETED with completed_at timestamp
    - Delete workflow Value Properties (PERS-04)
    - Release all workflow resources via resource manager
    - Destroy workflow resource pools
    - Log WORKFLOW_COMPLETED event

    `abortWorkflow(workflowInstanceId, activeSteps, repositories)`:
    - Send ABORT to all active steps' state machines
    - Release all resources
    - Set workflow state to ABORTED
    - Log WORKFLOW_ABORTED event

    **Create ExecutionLogService** (`src/logger/execution-logger.ts`):
    - Constructor takes IExecutionLogger (repository interface)
    - `logEvent(workflowInstanceId, stepOid, stepInstanceId, eventType, eventData)`: Promise<void>
      - Create ExecutionLogEntry with ISO 8601 UTC timestamp
      - Serialize eventData to JSON string
      - Call logger.log(entry)
    - Logs EVERYTHING unconditionally per CONTEXT.md decisions
    - Log types from types/events.ts: LogEventType
    - Each entry captures: event_type, previous state, new state, timestamp, and related data (full audit trail)

    **Create StepExecutor** (`src/runner/step-executor.ts`):
    Step-type-specific execution logic. Called by WorkflowRunner when a step enters each phase.

    `executeStartingPhase(step, context)`:
    - Resolve input parameters via ParameterResolver.resolveInputs()
    - Store resolved inputs in step (resolved_inputs_json)
    - Log PARAMETER_INPUT_RESOLVED events
    - Transition step: SC (STARTING -> EXECUTING)

    `executeExecutingPhase(step, context)`:
    - Based on step_type:
      - **START**: Auto-complete immediately (SC -> COMPLETING). START steps have no user interaction.
      - **END**: Auto-complete immediately (SC -> COMPLETING). Triggers workflow completion.
      - **USER_INTERACTION**: Emit USER_INPUT_REQUIRED event. Step stays in EXECUTING until user submits.
      - **YES_NO**: Emit USER_INPUT_REQUIRED event. Step stays in EXECUTING until user submits yes/no.
      - **SELECT_1**: Resolve the input value, evaluate conditions via ConditionEvaluator, store matched connection. Then SC -> COMPLETING.
      - **PARALLEL**: Auto-complete immediately (SC -> COMPLETING). The scheduler handles branch activation.
      - **WAIT_ALL**: Auto-complete immediately (SC -> COMPLETING). The scheduler already verified all branches are done.
      - **WAIT_ANY**: Auto-complete immediately (SC -> COMPLETING). The scheduler already verified at least one branch is done.
      - **WORKFLOW_PROXY**: Defer to Phase 4 implementation. For now: throw UnsupportedStepTypeError("WORKFLOW_PROXY requires Phase 4 implementation").
      - **ACTION_PROXY**: Throw UnsupportedStepTypeError("ACTION_PROXY steps require action server configuration (v2 feature)")
      - **SCRIPT**: Throw UnsupportedStepTypeError("SCRIPT steps require Pyodide (v2 feature)")

    `executeCompletingPhase(step, context)`:
    - Write output parameters via ParameterResolver.writeOutputs()
    - Log PARAMETER_OUTPUT_WRITTEN events
    - Transition step: SC (COMPLETING -> COMPLETED)
    - Notify scheduler of completion (returns next steps to activate)

    **Create WorkflowRunner** (`src/runner/workflow-runner.ts`):
    The top-level orchestrator. All operations go through the serial EngineEventQueue.

    Constructor takes RunnerConfig. Stores active workflow state in a Map<string, WorkflowRunnerState>.

    `createWorkflow(masterSpec)`: Promise<string>
    - Call lifecycle.createRuntimeWorkflow()
    - Save workflow, steps, connections to repositories
    - Initialize Value Properties from spec defaults
    - Initialize resource pools
    - Build scheduler adjacency lists
    - Create StateMachine instances for each step
    - Store state in active workflows map
    - Log WORKFLOW_CREATED
    - Return workflow instance_id

    `startWorkflow(workflowInstanceId)`: Promise<void>
    - Update workflow state to RUNNING
    - Find START step
    - Enqueue activation event for START step through event queue
    - Log WORKFLOW_STARTED

    `activateStep(stepInstanceId)` (internal, called via event queue):
    - Transition step: START (IDLE -> WAITING)
    - Check resource commands (if any):
      - Acquire resources in alphabetical order
      - If all acquired: transition SC (WAITING -> STARTING)
      - If any queued: step stays in WAITING until resources granted
    - If no resource commands: immediately transition SC (WAITING -> STARTING)
    - Execute STARTING phase (resolve inputs)
    - Execute EXECUTING phase (step-type-specific)
    - If step auto-completes (START, END, PARALLEL, WAIT_ALL, WAIT_ANY, SELECT_1): proceed to COMPLETING
    - If step waits for user (USER_INTERACTION, YES_NO): stay in EXECUTING

    `submitUserInput(stepInstanceId, formData)`: Promise<void>
    - Store formData as user_inputs_json on step
    - Log USER_INPUT_SUBMITTED with form data (for PDF audit trail)
    - Enqueue event: transition step SC (EXECUTING -> COMPLETING)

    `onStepCompleted(stepInstanceId)` (internal, called when step reaches COMPLETED):
    - Log STEP_STATE_CHANGED
    - Get scheduler next steps
    - Log SCHEDULER_DECISION with activated step IDs
    - For each next step: enqueue activation
    - If completed step is END: call completeWorkflow

    `pauseWorkflow(workflowInstanceId)`: Pause all EXECUTING steps
    `resumeWorkflow(workflowInstanceId)`: Resume all PAUSED steps
    `abortWorkflow(workflowInstanceId)`: Abort via lifecycle
    `stopWorkflow(workflowInstanceId)`: Stop via lifecycle (orderly shutdown)

    **CRITICAL: All step state changes and activations go through the EngineEventQueue.** This prevents race conditions when parallel branches complete simultaneously (Research Pitfall 1). The event queue processes one event fully before dequeuing the next.

    **Create crash recovery** (`src/runner/crash-recovery.ts`):

    `recoverWorkflows(config: RunnerConfig)`: Promise<RecoveryResult>
    - Query active workflows: repository.getActive() (workflow_state NOT IN COMPLETED, ABORTED, STOPPED, IDLE)
    - For each active workflow:
      1. Load all steps, connections
      2. Build scheduler adjacency lists
      3. Create StateMachine instances (initialize to current persisted state, not IDLE)
      4. For each step in non-terminal state, apply per-state recovery:

         | State at Crash | Recovery Action |
         |----------------|----------------|
         | IDLE | No action |
         | WAITING | Re-enter WAITING (retry resource acquisition) |
         | STARTING | Roll back to WAITING (re-resolve inputs) |
         | EXECUTING (USER_INTERACTION/YES_NO) | Stay EXECUTING (re-display form) |
         | EXECUTING (auto-complete types) | Re-execute and send SC |
         | COMPLETING | Re-execute COMPLETING and send SC |
         | PAUSING | Complete to PAUSED |
         | PAUSED | Stay PAUSED |
         | UNPAUSING | Complete to EXECUTING |
         | HOLDING/HELD/UNHOLDING | Stay in current state |
         | ABORTING | Complete to ABORTED |
         | STOPPING | Complete to COMPLETED |
         | COMPLETED/ABORTED | No action (terminal) |

      5. Check stale threshold: if last_activity_at is more than 24 hours ago, flag as stale
      6. For non-stale workflows: resume normal execution (re-activate steps that need processing)
      7. For stale workflows: return in stale list (UI layer will prompt user to resume or abort)

    - Return RecoveryResult: { recovered: string[], stale: string[], errors: Array<{ workflowId: string, error: string }> }

    `RecoveryResult` type in `src/runner/types.ts`.
