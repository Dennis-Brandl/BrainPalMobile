### Missing Connections (2)

**1. Crash recovery: stepsToReactivate discarded**
- File: `packages/engine/src/runner/crash-recovery.ts`
- `recoverWorkflows()` builds a `stepsToReactivate` array for steps in WAITING/STARTING/COMPLETING states, but the array is only used in a log message and never returned in `RecoveryResult`
- Impact: After app crash, non-user automated steps (START, PARALLEL, WAIT_ALL, SELECT_1) that were mid-flight are left frozen. USER_INTERACTION/YES_NO steps in EXECUTING recover correctly because execution store finds them via state query.
- Severity: Low — automated steps execute in milliseconds, so the crash window is tiny. Long-lived steps are user interaction steps which recover correctly.

**2. WORKFLOW_PAUSED event not emitted**
- File: `packages/engine/src/runner/workflow-runner.ts`
- `pauseWorkflow()` updates DB to PAUSED but emits no event. No `WORKFLOW_PAUSED` event exists in `EngineEventMap`.
- Impact: After pausing, Zustand store's `workflowState` stays RUNNING. StateControls shows Pause button instead of Resume. DB state is correct, and navigating away/back would re-sync from DB.
- Severity: Medium — UX bug, not data integrity. Workaround: navigate away and back.
