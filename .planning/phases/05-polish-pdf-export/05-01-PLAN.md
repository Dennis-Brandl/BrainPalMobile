    **1. Install expo-print, expo-sharing, expo-file-system:**

    **2. Create `apps/mobile/src/services/pdf-report-service.ts` (mobile):**
    Build a pure function `buildReportHtml(data: ReportData): string` that generates a complete HTML document for the PDF. The HTML must:
    - Include `<!DOCTYPE html>` to avoid extra blank page on iOS (Pitfall 1 from RESEARCH.md)
    - Use `@page { size: A4 portrait; margin: 20mm; }` CSS
    - Set `body { margin: 0; font-family: 'Helvetica Neue', Arial, sans-serif; color: #111827; }`
    - **Cover page**: centered layout with `page-break-after: always`, workflow name (h1, 28px), "Execution Report" subtitle, table showing Run ID (first 8 chars of instanceId), Date (formatted startedAt), Outcome (colored: #10B981 for COMPLETED, #EF4444 for ABORTED/STOPPED), Duration
    - **Step Summary section**: h2 header "Step Summary" with blue (#2563EB) bottom border, then a table with columns: Step, Outcome, Duration, Details
    - Each step row shows: step name, outcome (state with color: #10B981 for COMPLETED, #EF4444 for ABORTED, #6B7280 for others), duration, and details (user inputs parsed from userInputs JSON as "field: value" pairs, or branching decision from resolvedOutputs JSON showing matchedConnectionId)
    - **Nested child workflow steps**: Group steps by checking if they come from a child workflow. The `useWorkflowHistory` query already includes child workflow steps via the JOIN. Add an indent style (padding-left: 20px) for child steps. To identify child steps, add a `parentWorkflowId` field to the query (see Task 2 for the data shape). If a step's workflow is a child, render it with indent and a small "Child: {childWorkflowName}" label.
    - **Footer**: fixed position at bottom `Generated by BrainPal` in 10px #6B7280 text
    - All text content must be HTML-escaped (write a simple `escapeHtml` function for &, <, >, ", ')

    Export an `exportPdf(data: ReportData)` async function that:
    1. Calls `buildReportHtml(data)` to get HTML
    2. Calls `Print.printToFileAsync({ html, width: 595, height: 842 })` to generate PDF
    3. Renames the file to `{WorkflowName}_run-{runId}.pdf` using expo-file-system. Try the new File API first: `import { File, Paths } from 'expo-file-system/next'`. If the SDK 54 API is `expo-file-system` (not `/next`), adjust accordingly. The key operation is moving from the printToFileAsync output URI to a cache file with the desired name. If the new File API proves problematic, fall back to the legacy `moveAsync` from `expo-file-system`.
    4. Calls `Sharing.shareAsync(destUri, { mimeType: 'application/pdf', UTI: 'com.adobe.pdf', dialogTitle: 'Share Execution Report' })`

    Define the `ReportData` interface:
    ```typescript
    export interface ReportData {
      workflowName: string;
      instanceId: string;
      state: string;
      startedAt: string | null;
      completedAt: string | null;
      duration: string;
      steps: ReportStep[];
    }

    export interface ReportStep {
      name: string;
      stepType: string;
      state: string;
      duration: string;
      userInputs: string | null;
      resolvedOutputs: string | null;
      isChildStep: boolean;
      childWorkflowName?: string;
    }
    ```

    **3. Create `apps/mobile/src/services/pdf-report-service.web.ts` (web variant):**
    Export the same `exportPdf(data: ReportData)` function signature. Implementation:
    1. Call `buildReportHtml(data)` (import shared HTML builder — OR duplicate inline since web doesn't have expo-print/sharing)
    2. Create a hidden iframe, write the HTML into it, call `iframe.contentWindow?.print()`
    3. Clean up iframe after 1 second timeout
    Note: `buildReportHtml` should be importable. If Metro resolves `.web.ts` for the whole module, the web variant must include its own copy of `buildReportHtml` or share it from a separate file. Simplest approach: duplicate `buildReportHtml` in the web file (it's a pure string template function, small enough). OR extract `buildReportHtml` into a shared `pdf-report-template.ts` file (no platform deps) imported by both. Prefer extraction to avoid duplication.

    Actually, **best approach**: Create `apps/mobile/src/services/pdf-report-template.ts` with the shared `buildReportHtml`, `ReportData`, `ReportStep`, and `escapeHtml` exports. Then both `pdf-report-service.ts` (mobile) and `pdf-report-service.web.ts` (web) import from this shared template file and only contain the platform-specific export logic.

    **4. Create `apps/mobile/src/hooks/useExportPdf.ts`:**
    ```typescript
    export function useExportPdf() {
      const [isExporting, setIsExporting] = useState(false);

      const exportReport = useCallback(async (data: ReportData) => {
        if (isExporting) return;
        setIsExporting(true);
        try {
          const { exportPdf } = await import('../services/pdf-report-service');
          await exportPdf(data);
        } catch (err) {
          console.warn('PDF export failed:', err);
          Alert.alert('Export Failed', 'Could not generate the PDF report.');
        } finally {
          setIsExporting(false);
        }
      }, [isExporting]);

      return { exportReport, isExporting };
    }
    ```
    Use dynamic import so Metro resolves the platform-specific file at runtime.
