    **2. Update step-repository.ts** (`apps/mobile/src/repositories/step-repository.ts`):

    - Add `child_workflow_instance_id` to the `save()` INSERT/ON CONFLICT statement (both column list and values)

    **3. Update lifecycle.ts** (`packages/engine/src/runner/lifecycle.ts`):
    - In `createRuntimeWorkflow`, add `child_workflow_instance_id: null` to the step creation in the `steps.map()` callback

    **4. Implement WORKFLOW_PROXY in step-executor.ts** (`packages/engine/src/runner/step-executor.ts`):
    - Replace the `throw new UnsupportedStepTypeError(...)` in the WORKFLOW_PROXY case
    - Add `workflowRepo` and `connectionRepo` to `StepExecutionContext` interface (needed for child workflow operations)
    - The WORKFLOW_PROXY case should:
      a. Parse the master step JSON to get the step spec
      b. Get the parent workflow's full specification_json from workflowRepo
      c. Parse the parent spec and find the matching child workflow from `child_workflows[]` array. The matching strategy: the WORKFLOW_PROXY step likely references the child by the step's own `local_id` or `description` matching a child workflow's `local_id`. Alternatively, if there is only one child workflow, use it. If multiple children exist, match by comparing step properties to child workflow identifiers (try `masterStep.local_id` against `childSpec.local_id`, or iterate and find by position/naming). Use a best-effort approach: if `child_workflows.length === 1`, use it directly; otherwise match by `masterStep.description` containing the child's `local_id`, or fall back to index-based matching
      d. Call `runner.createChildWorkflow(childSpec, workflowInstanceId, step.step_oid)` -- a new method on WorkflowRunner (see below)
      e. Save the `child_workflow_instance_id` on the parent step
      f. Start the child workflow via `runner.startWorkflow(childInstanceId)`
      g. Return `false` -- the parent step stays in EXECUTING while child runs

    **Note:** The step executor needs access to the WorkflowRunner. Add an optional `runner?: WorkflowRunner` field to `StepExecutionContext` (only set for WORKFLOW_PROXY steps). Import WorkflowRunner type only (avoid circular dependencies by using the interface pattern or passing the runner through context).

    IMPORTANT: To avoid circular import between step-executor.ts and workflow-runner.ts, define a `IWorkflowRunnerForProxy` interface in step-executor.ts (or runner/types.ts) with just the methods needed: `createChildWorkflow(spec, parentId, parentStepOid)` and `startWorkflow(id)`. WorkflowRunner implements this interface. Pass the runner through StepExecutionContext as `runner?: IWorkflowRunnerForProxy`.

    **5. Add `createChildWorkflow` method to WorkflowRunner** (`packages/engine/src/runner/workflow-runner.ts`):
    - New method `async createChildWorkflow(childSpec: MasterWorkflowSpecification, parentWorkflowInstanceId: string, parentStepOid: string): Promise<string>`:
      a. Call existing `createWorkflow(childSpec)` to get the child instance ID
      b. Fetch the child workflow from workflowRepo and set `parent_workflow_instance_id = parentWorkflowInstanceId` and `parent_step_oid = parentStepOid`
      c. Save the updated child workflow
      d. Return the child instance ID

    **6. Add child completion -> parent resumption logic** (`packages/engine/src/runner/workflow-runner.ts`):
    - In `onStepCompleted`, when an END step completes, BEFORE calling `completeWorkflow`:
      - Check if the completing workflow has a `parent_workflow_instance_id`
      - If yes, this is a child completing:
        a. Complete the child workflow (existing completeWorkflow call)
        b. Emit WORKFLOW_COMPLETED for the child
        c. Retrieve the parent's runner state from `activeWorkflows`
        d. Find the parent step (the WORKFLOW_PROXY step) using the child's `parent_step_oid`
        e. Get the parent step's instance_id from `stepOidToInstanceId`
        f. Propagate child output parameters: read child workflow's output Value Properties and store them in the parent step's `resolved_outputs_json`
        g. Directly call `handleUserInputCompletion` for the parent step (EXECUTING -> COMPLETING). Use direct call, NOT eventQueue.enqueue, because we are already inside the event queue handler. This avoids Pitfall 3 from research.
        h. Remove child from activeWorkflows
        i. Return early (do NOT process parent workflow scheduling -- that's handled by the parent step's completion)
      - If no parent, proceed with existing logic (emit WORKFLOW_COMPLETED, delete from activeWorkflows)

    **7. Add pause/abort/resume propagation** (`packages/engine/src/runner/workflow-runner.ts`):
    - In `pauseWorkflow`: After pausing each step, if `step.step_type === 'WORKFLOW_PROXY'` and `step.child_workflow_instance_id`, check if child workflow is RUNNING and recursively call `this.pauseWorkflow(step.child_workflow_instance_id)`
    - In `resumeWorkflow`: Similarly, if step was WORKFLOW_PROXY with a child, recursively call `this.resumeWorkflow(step.child_workflow_instance_id)`
    - In `abort`: Similarly, if step is WORKFLOW_PROXY with a child, recursively call `this.abort(step.child_workflow_instance_id)` BEFORE aborting the parent step (so child is cleaned up first)
    - In `stop`: Similarly propagate to child

    **8. Wire runner into StepExecutionContext** (`packages/engine/src/runner/workflow-runner.ts`):
    - Update `buildStepContext` to include `runner: this` (as IWorkflowRunnerForProxy) and `workflowRepo: this.config.workflowRepo` and `connectionRepo: this.config.connectionRepo` in the context
    - Update `StepExecutionContext` type accordingly

    **9. Handle child_workflow_instance_id in step save/load**:
    - The step repo `save()` must now include child_workflow_instance_id in both the INSERT column list and ON CONFLICT SET clause
    - The `rowToStep` must map the column from the row
