**1. Create useActiveSteps hook** (`apps/mobile/src/hooks/useActiveSteps.ts`):

Derives the list of active user interaction steps for the carousel from the execution store.

```typescript
interface ActiveStep {
  stepInstanceId: string;
  stepOid: string;
  stepType: StepType;
  stepState: StepState;
  stepSpec: MasterWorkflowStep;  // parsed from step_json
  formLayout: FormLayoutEntry | null;  // selected for current device type
}

function useActiveSteps(workflowInstanceId: string): {
  steps: ActiveStep[];
  isLoading: boolean;
}
```

Implementation:
1. Read from execution store: `activeWorkflows[workflowInstanceId]` to get `activeStepInstanceIds`
2. For each active step instance ID, load the RuntimeWorkflowStep from SQLite (use `useSQLiteContext()` to query `runtime_steps WHERE instance_id = ?`)
3. Filter to only USER_INTERACTION and YES_NO step types (these are the steps that need form rendering)
4. Parse `step_json` from each step to get the `MasterWorkflowStep` spec
5. Select the appropriate `FormLayoutEntry` from `form_layout_config` based on `useDeviceType()`:
   - Try to find entry matching current device type ('phone', 'tablet', 'desktop')
   - Fall back to first available entry if no match
   - Return null if no form_layout_config exists
6. Return the list of ActiveStep objects

**Use `useMemo` or `useEffect` + state** to avoid recalculating on every render. Re-derive when `activeStepInstanceIds` changes (from execution store).

**Pitfall 2 from research:** Carousel position can desync with step list during parallel execution. Mitigate by using stable keys (stepInstanceId) and re-deriving index when the list changes.

**2. Create StepCarousel** (`apps/mobile/src/components/carousel/StepCarousel.tsx`):

FlatList-based horizontal carousel with wrap-around navigation.

Props:
```typescript
interface StepCarouselProps {
  steps: ActiveStep[];
  currentIndex: number;
  onIndexChange: (index: number) => void;
  onStepComplete: (stepInstanceId: string, outputValue?: string) => void;
  images: Map<string, string>;  // image filename -> base64 URI
}
```

Implementation:
1. Horizontal `<FlatList>` with `pagingEnabled={true}`, `showsHorizontalScrollIndicator={false}`
2. `getItemLayout` for consistent page sizing (each item = screen width)
3. `keyExtractor` using `step.stepInstanceId` (stable ID)
4. Each page renders:
   - Step name/title (from stepSpec.local_id)
   - `<FormCanvas>` with the step's formLayout, form data state, and onChange handler
   - `<FormActionButtons>` with appropriate type (USER_INTERACTION or YES_NO) and yes_no_config
5. `onViewableItemsChanged` callback to update `currentIndex` when user swipes
6. Previous/Next buttons:
   - Fixed position at left/right edges of the carousel (or below)
   - Previous: `scrollToIndex(currentIndex - 1)` with wrap-around (if index < 0, go to `steps.length - 1`)
   - Next: `scrollToIndex(currentIndex + 1)` with wrap-around (if index >= steps.length, go to 0)
   - Use `FlatList.scrollToIndex({ index, animated: true })`
   - Disable or hide if only 1 step
7. Form data state: Each step page maintains local form data in a `Record<string, string>`. Use a `Map<stepInstanceId, Record<string, string>>` at the carousel level to persist form data when swiping between steps.
8. On step complete (FormActionButtons onSubmit):
   - Call `onStepComplete(stepInstanceId, outputValue)` which will call WorkflowRunner.submitUserInput
   - After completion, auto-advance: if there are more active steps, scroll to the next one. If the completed step was the last one and no more steps are active, the carousel will be empty (show waiting state).

The main execution screen that brings everything together.

Route parameter: `instanceId` (runtime workflow instance ID).

Implementation:

1. Get `instanceId` from `useLocalSearchParams()`
2. Get engine runner from `useEngine()`
3. Get execution store state for this workflow from `useExecutionStore()`
4. Get active steps from `useActiveSteps(instanceId)`
5. State:
   - `currentStepIndex` (number)
   - `showAbortConfirm` (boolean)
   - `formDataMap` (Map<string, Record<string, string>>) -- form data per step
   - `images` (Map<string, string>) -- loaded step images

6. Load images on mount:
   - Query package_images table for the workflow's master_workflow_oid
   - Convert BLOB data to base64 data URIs
   - Store in images Map
   - Cache in memory (Pitfall 4 from research)

7. Layout:
   ```
   SafeAreaView (flex: 1)
     ExecutionHeader
       workflowName (from execution store or parsed spec)
       workflowState
       stepInfo ("Step {currentIndex+1} of {steps.length}")
       StateControls as stateControlsSlot

     {steps.length > 0 ? (
       <>
         StepCarousel
           steps={activeSteps}
           currentIndex={currentStepIndex}
           onIndexChange={setCurrentStepIndex}
           onStepComplete={handleStepComplete}
           images={images}
         DotIndicator
           count={steps.length}
           activeIndex={currentStepIndex}
       </>
     ) : (
       WaitingStateBox
         workflowState={workflowState}
     )}

     ConfirmDialog (for abort)
   ```

8. Event handlers:

   **handleStepComplete(stepInstanceId, outputValue?)**:
   - Get form data from formDataMap for this step
   - If YES_NO step: merge outputValue into form data as the output parameter
   - Call `runner.submitUserInput(stepInstanceId, formData)`
   - Auto-advance: wait for execution store to update activeSteps, then move to next step in carousel
   - If no more active steps, carousel hides and WaitingStateBox shows

   **handlePause**:
   - Call `runner.pauseWorkflow(instanceId)`

   **handleResume**:
   - Call `runner.resumeWorkflow(instanceId)`

   **handleStop**:
   - Call `runner.abort(instanceId)` with stop semantics (the engine's abort method handles stopping active steps)
   - Actually, check engine API: the runner has `pauseWorkflow` and `abort`. For STOP behavior (which can be resumed), use `pauseWorkflow`. For ABORT (terminal), use `abort`. If the engine doesn't have a separate stop method, STOP = pause + set state to STOPPED. Check the WorkflowRunner API and adapt.

   **handleAbort**:
   - Show ConfirmDialog
   - On confirm: call `runner.abort(instanceId)`
   - On confirm: navigate back to home (`router.back()`)

9. Auto-advance behavior (from CONTEXT):
   - After step completion, when new active steps arrive via execution store update, if the completed step was at index N, try to show index N (next step slides in). If N >= new steps.length, show 0.
   - Use a `useEffect` watching `activeSteps` to handle index adjustment when steps change.

10. Handle workflow completion:
    - When workflowState becomes 'COMPLETED' or 'ABORTED', show a completion overlay or navigate back to home after a brief delay.

**Important -- parallel execution support:**
When PARALLEL fork activates multiple branches, the engine emits ACTIVE_STEPS_CHANGED with all active step IDs. The useActiveSteps hook filters to USER_INTERACTION and YES_NO types. Multiple active UI steps appear as multiple pages in the carousel. The user swipes between them. This naturally supports parallel branches without any special parallel-specific UI code -- the carousel just shows whatever steps are active.

For WAIT_ALL and WAIT_ANY joins, the engine handles the logic internally. When all/any branches complete, the join step proceeds automatically. If the join step is a UI step, it appears in the carousel. If it's an automatic step, it processes and the next step appears.

Resources (EXEC-08) and SYNC barriers (EXEC-09) are handled by the engine. If a step is waiting for a resource, it stays in WAITING state (not EXECUTING), so it won't appear as an active UI step. The WaitingStateBox handles this case.
